Map an existing or new project into Anchor's responsibility model without unnecessary migration.

---

## What this practice is

Project orientation answers: *Who owns what knowledge in this project?*

It does not require rewriting existing ADRs, architecture docs, or AI context files. It maps responsibilities, fills gaps, and trims overlap.

---

## When to use each expression

**Start with Ch. 4** if you are adopting Anchor for the first time. It explains what you will have at the end and helps you choose new project vs existing project.

**Follow the playbook** when you are ready to execute. The playbook prescribes order: assess → classify → orient → review → handoff.

**Run the review** after new contract files exist. Pass/fail only — no teaching.

**Study the journey** when you want to compare your repo to a worked example across four sessions.
